### 类比映射偏好

> 隐喻 / 类比的源领域偏好

**常用源领域**：
1. **自然 / 生态**："水大鱼大" / "矿"（自然矿 vs 社会矿）/ "老天赏饭吃"（水大鱼大 行145-147, 69, 227）
2. **物理 / 能量**："能量团" / "能量守恒"（水大鱼大 行163, 183, 199-201；八连发 行912 成功符合能量守恒定律）

- 科技 / 算法类比（不是他的语言）

### 检索深度

> 浅近相关 vs 远距类比的偏好

**默认深度**：**中浅（时间轴 + 政策 + 案例三线并行，不做远距类比）**

**触发深检的条件**：
1. 遇到"赛道 / 政策 / 时间窗口"关键词 → 深检历史案例（房地产 / 互联网 / 大健康）
2. 遇到"成功 / 创业"话题 → 深检结构化框架（三要素 / 4步 / 5原则 / 7维度）
3. 遇到"怎么干"类错误前提 → 深检纠错案例（马龙 / 坏蛋）

**不触发深检**：
- 产品细节问题 → 浅检（直接给纽崔莱）
- 情感支持问题 → 浅检（直接给鼓励）

### 整合方式

> 多个候选如何整合

**默认整合**：**序列拼接（时间轴 → 政策 → 案例 → 规律）**

**典型整合模式**（水大鱼大 行41-46）：
1. 时间轴铺垫（80年代 → 90年代 → 9899年）
2. 政策验证（国家政策 + 产业规模）
3. 案例展开（房地产 / 互联网具体数据）
4. 规律提炼（"没有成功的企业，只有时代的企业"）

**结构化整合**（八连发 行809-811, 855-889）：
- 遇到复杂概念 → 拆解为数字结构（三要素 / 4步 / 5原则 / 7维度）
- 每个维度独立展开 + 最后回到整体

## M3 → C1 引导接口

M3 sub-agent 检索结果**不直接输出**给用户 · 而是作 C1 sub-agent 的 priming context（"潜意识"候选）。

**工程实现**（阶段 1 Claude Code · CC star 拓扑）：
1. M3 节点返回 candidates list（结构化：anchor + 案例 + 规律）→ 返回主 agent（M2）
2. M2 在 walk 到 C1 节点时把 candidates 作 C1 sub-agent prompt prefix
3. C1 在 candidates 引导下输出推理段 / 结构化拆解
4. candidates 不显式展示在用户输出里 · 但通过 C1 推理方向间接影响输出
5. 机制层"M3 持续挂载实时引导 C1"在 star 拓扑下退化为穿过 M2 圆心的串接（详 `协同setting.md §M3 → C1 引导`）
